# Integration entities are canonical, not per-source

The `integration` schema holds one `DW_<TABEL>` per business entity, regardless of how many source systems contribute rows. KRM names this layer "Combine" because consolidating cross-source entities is its purpose. Per-row provenance lives in the `WA_SRC` admin column (which source system that row's CDF event originated from), not in the table name. If two source systems contribute to the same business entity — for example `ms_crm.klant` and `azurestorage.klant` both representing customer records — both feed the same `DW_KLANT`, with `WA_SRC` distinguishing per row, and the per-entity DLT SQL file owns the cross-source UNION and any BK-alignment logic (consistent with ADR-0009, which locates entity structure in the SQL that creates the entity).

The alternative — naming integration tables `DW_<TABEL>_<SRC>` to keep sources segregated — was considered and rejected. It defeats KRM's "Combine" purpose, forces every downstream consumer to choose a source (or write their own UNION), and turns the datamart layer into a per-source-times-per-entity matrix instead of the entity-canonical star the KRM "Publish" layer expects. The cost of canonical consolidation — entity-matching logic when source systems disagree on identity — is real, but it lives in the per-entity SQL where it can be reviewed alongside the rest of the entity's transformation; it does not need to be solved generically up front. The current demo has no collision (only Azure Storage is active); this ADR sets the policy for the case when SQL Server is unparked or new sources arrive.
