# SCD1 dim views project from the latest DWH row per BK

For each Type-1 dimension `DIM_<NAAM>` in the `datamart` schema, the view selects exactly one row per business key — the latest row in `DWH_<TABEL>` by `WA_FROMDATE`, not the row where `WA_ISCURR = 1`. The latest-row choice is what lets a deleted entity remain visible in the SCD1 dimension; under ADR-0010's `APPLY CHANGES INTO STORED AS SCD TYPE 2` mechanism, deletes set `__END_AT` on the previously-current row, leaving no `WA_ISCURR = 1` row for that BK at all. The surrogate `MK_<TABEL>` is the lineage's `WKR_<TABEL>` — the root surrogate, first version ever assigned to this BK — making `MK_` stable across all future updates and deletes of the entity. `MA_CREATEDATE` is `MIN(WA_CRUDDTS) PARTITION BY <BK>`; `MA_CHANGEDATE` is `MAX(WA_CRUDDTS) FILTER (WHERE WA_CRUD <> 'C') PARTITION BY <BK>`, NULL when the BK only has its initial C row — so updates *and* deletes both contribute to "last changed"; `MA_ISDEL` is 1 when the latest row's `WA_UNTODATE` is anything other than `'9999-12-31'` (i.e., the lineage has been end-dated/deleted).

This spec corrects three formulas from naamgeving §2.6. Naamgeving's `MK_<TABEL>` ← `WK_<TABEL>` is rejected because the latest version's surrogate changes on every update, defeating the purpose of a stable datamart key; KRM slide 15 uses the root surrogate and we follow KRM. Naamgeving's `MA_CHANGEDATE` filter `WHERE WA_CRUD = 'U'` is rejected because it ignores deletes — KRM slide 15 shows the deleted entity's `MA_CHANGEDATE` is the delete date, not the prior update date. Naamgeving's "alleen `WA_ISCURR = 1`" source filter is rejected because under ADR-0010's SCD2 mechanism it silently drops deleted entities from the dimension; the latest-row-per-BK selection preserves them.
