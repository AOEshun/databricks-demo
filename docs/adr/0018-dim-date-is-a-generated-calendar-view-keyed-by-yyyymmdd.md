# DIM_DATE is a generated calendar view keyed by yyyymmdd integer

`DIM_DATE` in the `datamart` schema is a view, not a materialised dimension and not derived from any `DW_` table — date is intrinsically version-less, so the historical-layer pattern does not apply. The view is generated dynamically each pipeline run via `SEQUENCE(MIN(<fact date>), MAX(<fact date>))` exploded into one row per date, covering exactly the date range observed in the fact data and growing automatically as new days arrive. The surrogate `MK_DATE` is the date itself encoded as an integer in `yyyymmdd` form (e.g. 2023-01-29 → 20230129), following KRM slide 14 — no SHA2 hash and no IDENTITY surrogate, because the date value is already deterministic, lexicographically sortable as an integer, and recognisable at a glance. The view exposes the standard calendar attribute set used by current code: `full_date DATE`, `year` / `quarter` / `month` / `day` INT, `month_name` / `day_name` STRING, `day_of_week` / `week_of_year` INT, `is_weekend` BOOLEAN, and `year_month_start` / `year_quarter_start` / `year_start` DATE for time-bucket roll-ups. A holiday calendar is not in scope.

Multiple date roles on the same fact — `order_ts`, `served_ts`, planned billing date, etc. — all reference the same `DIM_DATE` through their own role-named FK columns on the fact (`MK_ORDERDATE`, `MK_SERVEDDATE`, …), exactly as naamgeving §5.5 illustrates with `MK_BOEKDAT`. A per-role dimension (`DIM_ORDERDATE`, `DIM_SERVEDDATE`) was considered and rejected: the underlying calendar is identical, so per-role tables would just be copies, and the consumer's join syntax is already clear from the FK column name without an aliased dim. The fixed-range alternative — generating, say, 2010-01-01 through 2050-12-31 regardless of data — was also rejected: it wastes rows for unused dates and silently allows facts to reference dates outside the observed business range without surfacing the anomaly.
