# DWH views carry no self-side BK hash column

The KRM-defined view-only column `WKR_HASH_<TABEL>` (slide 10 — hash of the lineage's business key) is not added to DWH views in this project. KRM exists in a world where identity surrogates differ across environments, so a deterministic hash of the natural BK is useful as an environment-stable identifier and as an anonymized join key. Neither use case applies here: the single `DEMO` catalog is the same across dev/test/prod targets, surrogate sequences align, no anonymization requirement is in scope, and the natural BK is already a passthrough column on every DWH view. The FK-side hash `WK_REF_HASH_<REF>` is kept because it bridges the identity-surrogate model to deterministic FK matching at fact-build time — a real, used capability — but the self-side `WKR_HASH_` would be a column without a consumer. We accept the asymmetry: hash on the FK side, no hash on the self side.
