# SCD2 dim views expose every DWH version, grouped by MK_ROOT

For each Type-2 dimension `DIM_<NAAM>` in the `datamart` schema, the view exposes every version of the entity — one datamart row per row in `DWH_<TABEL>`. `MK_<TABEL>` carries forward the version-level surrogate `WK_<TABEL>`; `MK_ROOT` carries forward `WKR_<TABEL>`, so all versions of one business-key lineage share an `MK_ROOT` and consumers can group versions by it. Validity-period columns rename through: `MA_FROM` ← `WA_FROMDATE`, `MA_UNTO` ← `WA_UNTODATE`, `MA_ISCURR` ← `WA_ISCURR`. Business columns pass through unchanged. This matches KRM slide 14 and naamgeving §2.6 SCD2 exactly — no corrections needed, the ADR pins the spec.

One mechanism-driven caveat applies under ADR-0010: deleted entities have no `MA_ISCURR = 1` row, because `APPLY CHANGES INTO STORED AS SCD TYPE 2` records deletes by setting `__END_AT` on the previously-current row rather than writing a tombstone. A consumer asking "is this entity deleted?" answers it by checking that the latest row per `MK_ROOT` has `MA_UNTO <> '9999-12-31'`, matching the SCD1 convention in ADR-0012. KRM slide 14's tombstone-row layout for deleted entities is not reproduced verbatim — the same trade-off ADR-0010 already documented at the DW level applies here at the datamart level.
