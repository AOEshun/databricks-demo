# Entity structure lives in the SQL that creates it

There is no separate machine-readable registry of which entities exist, what columns they carry, or how they reference each other. The DLT SQL files in `integration/` and `datamart/`, together with the dim view notebooks in `views/datamart/`, are the canonical source of every entity's structural facts: business keys, column lists and types, FK references, datamart inclusion. Unity Catalog provides the queryable surface over those structures — schemas, lineage, table comments — without a separate inventory file. `CONTEXT.md` keeps a human-readable tabelspec section per entity so a reader can scan the inventory without grepping SQL, but it is documentation, not a source-of-truth; the SQL is authoritative and CONTEXT.md follows.

Two alternatives were considered and rejected. A `DEMO.CONFIG.entities` Delta control table mirroring every entity's structure would either duplicate facts already declared in the SQL (drift risk) or be generative — and a YAML-driven generative path is exactly the approach we explicitly stepped away from (see `naamgeving-en-lagen.md`). For *operational* decisions — which sources are active, in which mode — control tables remain the right pattern (see ADR-0004 as amended).
