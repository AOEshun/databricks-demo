# Pipeline topology: Workflow over per-layer DLT pipelines, dims as plain UC views

The end-to-end pipeline is a single Databricks Workflow (`resources/demo_workflow.yml`) orchestrating: a `setup` notebook (catalog/schema/volume/control-table bootstrap), parallel Auto Loader ingest notebook tasks (`ingest_azurestorage_*`, mode-widget driven from the control table per ADR-0004), the `dlt_integration` DLT pipeline covering each integration entity's tagged source MV + `DW_<TABEL>` + `DWQ_<TABEL>` + `DWH_<TABEL>`, the `dlt_datamart` DLT pipeline covering fact materialised views `FCT_<NAAM>` (with Liquid Clustering where volume justifies), and an `apply_views` notebook task that creates `DIM_<NAAM>` as plain Unity Catalog views over `DWH_<TABEL>` for consumer surfaces (dashboards, AI/BI Genie). Lakeflow Connect runs as its own separately-managed pipeline against `staging_sqlserver`; the Workflow does not orchestrate it. Facts in `dlt_datamart` read from `DWH_<TABEL>` directly — projecting `WKR_<TABEL>` as the `MK_<TABEL>` FK — rather than from `DIM_<NAAM>`, so the fact build is independent of the consumer dim views; `dlt_datamart` and `apply_views` can therefore run in parallel after `dlt_integration` completes. The Workflow's own task graph is itself a demo moment, showing the layer dependencies setup → ingest → integration → (datamart || apply_views).

Two alternative dim implementations were considered and rejected. DLT temporary views are pipeline-scoped and not queryable by dashboards or AI/BI Genie outside the pipeline. DLT materialised views are externally queryable but introduce refresh overhead unnecessary for what is effectively a rename projection over `DWH_`. A plain Unity Catalog view created by `apply_views` is zero-storage, always fresh, externally queryable, and matches current code's existing pattern; downstream performance optimisation for dashboards is a separate concern to be layered on later if measurements call for it. Two further deviations from upstream conventions are noted explicitly: naamgeving §1 ("*alles draait in Lakeflow Declarative Pipelines op serverless compute*") is relaxed for staging Auto Loader (control-table-driven, more idiomatic in imperative Python) and for the dim views; and Databricks' general guidance to combine silver and gold into one transformation pipeline is intentionally not followed here, since the integration-vs-datamart split surfaces the two layers as distinct DLT graphs and matches the demo's narrative.
