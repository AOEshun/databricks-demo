# Quality-failed CDF events route to a paired DWQ_<TABEL>

Every cleansed `DW_<TABEL>` in the `integration` schema is paired with an append-only quarantine streaming table `DWQ_<TABEL>` in the same schema. Both are populated by the same DLT pipeline from a shared intermediate "tagged" materialised view that computes a `failed_rules ARRAY<STRING>` column per CDF event — so the rule logic appears exactly once, in the tagged MV's `CASE WHEN` expressions. The tagged MV also carries `CONSTRAINT … EXPECT (NOT array_contains(failed_rules, '<rule>'))` declarations so each rule's per-run violation count surfaces in the DLT event log; downstream the cleansed source MV selects `WHERE size(failed_rules) = 0` and the quarantine selects `WHERE size(failed_rules) > 0`. A quarantine row carries every column from the rejected CDF event (including the raw `_change_type` for diagnostic purposes) plus `failed_rules`, `WA_CRUDDTS`, `WA_SRC` and `WA_RUNID` — no `WK_` surrogate, no `__START_AT`/`__END_AT`, no SCD2 housekeeping. Three severity levels survive from the current pattern: `warn` (DLT `EXPECT (…)` without `ON VIOLATION` — row stays in cleansed, violation logged), `drop` (routes via `failed_rules` into `DWQ_<TABEL>`), `fail` (`EXPECT (…) ON VIOLATION FAIL UPDATE` — pipeline halts).

Two alternatives were considered and rejected. Putting any quality checks on `STG_` tables (naamgeving §6.3 with `ON VIOLATION DROP ROW`) silently destroys rejected rows (forbidden by ADR-0007) and fragments the data-quality story across two layers; staging stays raw, with true load-time issues like malformed parquet captured by Auto Loader's `_rescued_data` column. The literal Databricks-recommended quarantine pattern — a Python pipeline using `@dp.expect_all(rules)` with a dict-driven `is_quarantined` boolean partition column — gives up two things we want to keep: SQL-first authorship (ADR-0003) and the per-row "which rule(s) did this row fail" diagnostic that makes demo-time triage SQL (`WHERE array_contains(failed_rules, 'order_total_non_negative')`) work. The tagged-MV-with-array SQL pattern preserves both at the cost of partition-pruning performance we don't need at demo scale.
