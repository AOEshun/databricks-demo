# All hash columns use SHA2-256

Every project-defined hash column is computed with `SHA2(<input>, 256)`. This covers `WA_HASH` (row-content signature, per ADR-0015), `WK_REF_HASH_<REF>` (deterministic hash of a referenced entity's natural business key, used at fact-build time to bridge the IDENTITY-surrogate model to source-system identifiers), and any future hash column introduced under the KRM admin-column conventions. Inputs are normalised before hashing: STRINGs are `COALESCE`'d to empty string, non-STRING types are `CAST` to STRING, and multi-column inputs are joined with `||` via `CONCAT_WS('||', …)`. The output is the 64-character lowercase hex digest Spark returns from `SHA2(…, 256)`.

Other digest functions (MD5, SHA-1, xxhash, CRC32) are not used. MD5 was considered for its smaller output footprint but rejected: column-storage in Delta is cheap, the 32-vs-64-byte saving per row is negligible at demo scale, and a single algorithm across the corpus avoids cross-ADR caveats and removes any "which hash did we use here?" ambiguity when reading SQL. Databricks-internal hashing (Liquid Clustering's bucketing function, partition-pruning hashes) is out of scope — that is engine-controlled and not a project decision.
