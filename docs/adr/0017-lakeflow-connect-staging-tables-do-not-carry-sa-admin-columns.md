# Lakeflow Connect staging tables do not carry SA_* admin columns

Source systems ingested via Lakeflow Connect (LC) — currently SQL Server, parked — produce staging tables that LC writes to the catalog directly with its own schema layout. The LC connector controls the columns and is not extensible; `SA_CRUDDTS` / `SA_SRC` / `SA_RUNID` cannot be added. LC tables are still placed in a `staging_<bron>` schema (e.g. `staging_sqlserver`) and named `STG_<TABEL>` where the connector allows, keeping the layer-and-naming convention consistent at the table level. What differs is the per-row admin: LC tables carry its native CDC metadata — `_change_type`, `_change_version`, `_commit_timestamp` — instead of the `SA_*` family. The integration layer adapts per source: for Auto Loader-sourced entities the per-entity DLT SQL maps `SA_CRUDDTS` → `WA_CRUDDTS` and uses staging CDF for sequencing; for LC-sourced entities the SQL maps `_change_type` → `WA_CRUD` and uses `_commit_timestamp` for sequencing directly. Both produce the same DW shape; the source-specific translation is contained in the per-entity SQL where it belongs (ADR-0009).

A view-based wrapper — `CREATE VIEW staging_sqlserver.STG_<TABEL> AS SELECT …, current_timestamp() AS SA_CRUDDTS, 'sqlserver' AS SA_SRC, NULL AS SA_RUNID FROM <lc_managed_table>` — was considered and rejected. It would hide the LC-vs-Auto-Loader asymmetry from the integration layer at the cost of inventing `SA_*` values that don't reflect anything real (e.g. `SA_CRUDDTS` would be the view-query time, not an ingestion timestamp), and the integration layer would still need to read `_change_type` for the `WA_CRUD` mapping since the view can't manufacture that. Materialising an `SA_*`-enriched copy of the LC table would violate ADR-0008 (one-time ingestion). The honest position is to acknowledge that LC manages its own staging shape and let the integration layer handle the per-source variance.
