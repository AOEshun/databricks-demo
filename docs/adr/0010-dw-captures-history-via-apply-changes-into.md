# DW captures change history via AUTO CDC INTO SCD TYPE 2

> **Status update (2026-05-15):** Mechanism renamed by Databricks (APPLY CHANGES INTO → AUTO CDC INTO / inline FLOW AUTO CDC). Binding architectural decision unchanged; clauses (KEYS / SEQUENCE BY / APPLY AS DELETE WHEN / STORED AS SCD TYPE 2) are identical.

The historical-layer DW tables in the `integration` schema are populated via Databricks' declarative `FLOW AUTO CDC ... STORED AS SCD TYPE 2`, reading from `STREAM table_changes()` over the source staging table's Change Data Feed. Business keys are passed as `KEYS`, sequencing is by `_commit_timestamp`, and source deletes are honoured via `APPLY AS DELETE WHEN _change_type = 'delete'`. Databricks manages the `__START_AT` and `__END_AT` columns; the paired DWH view renames them to `WA_FROMDATE` and `WA_UNTODATE`, derives `WA_ISCURR` from `__END_AT IS NULL`, and computes `WKP_<TABEL>` / `WKR_<TABEL>` via window functions over the business key.

The KRM-faithful alternative — an insert-only DW that records every CDF event as a separate row, including a tombstone row with blanked business columns for each delete (matching slide 11 of the KRM deck) — was considered and rejected. The 4-row pattern shown in KRM is illustrative of the SCD2 concept, not a literal mandate; bypassing `FLOW AUTO CDC` would surrender one of the most demo-worthy declarative DLT constructs in exchange for forensic detail that is recoverable via a simple "rows with `__END_AT IS NOT NULL` and no successor" query. We accept that the DW table does not visibly contain a dedicated D-event row.
