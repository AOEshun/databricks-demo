# WA_HASH is kept for source reconciliation, not change detection

Every `DW_<TABEL>` row carries a `WA_HASH` column computed as `SHA2(CONCAT_WS('||', <all non-BK business columns, COALESCE'd to empty string>), 256)`. The hash is computed in the tagged source MV before the APPLY CHANGES INTO step, so DWQ rows carry it as well — making rejected rows comparable too. Its purpose is **source-system reconciliation** (a downstream pipeline or one-off script can compute the same hash on the source and verify that DW reflects source content faithfully) and as a **forensic row signature** for audit-style queries. It is explicitly **not** used for change detection: ADR-0010's `APPLY CHANGES INTO STORED AS SCD TYPE 2` already compares key and tracked columns internally and only writes a new version when content changes, so a WA_HASH-based skip-update gate would be redundant.

Dropping WA_HASH was considered. The change-detection use case disappears under ADR-0010, and our single-catalog `DEMO` setup has no cross-environment reconciliation scenario, so the column has no current consumer. We keep it on three grounds: schema-migration avoidance (adding it later means rewriting history), demo familiarity for KRM-aware audience members who look for it, and future-extensibility for a reconciliation demo. Cost is one SHA2 call per row and a 64-byte STRING per DW row.
